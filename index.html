Margarita

Margarita templates bring structure and reusability to your prompts - renders to plain old markdown so you can use it anywhere you were already using markdown.

Explore Margarita Docs

Markdown Output

Anything between << and >> is plain Markdown

---
description: A template that demonstrates basic markdown formatting.
---
<<
# This is a normal heading
  - A list
A **bold** statement followwed by *italics* and a [link](https://www.banyango.com/margarita/latest).

Anything you can do in markdown you can do here.
>>
for item in items:
  << - This is also markdown -> ${item} >>

Variable Substitution

Pass data into your templates using ${variable} anywhere in the body.

# greet.mg
---
name: greet
description: A simple template that demonstrates variable substitution.
parameter1: name (string): The name of the person to greet.
parameter2: project (string): The name of the project or product.
---
<< Hello, ${name}! >>
<< You are using ${project}. >>

Conditionals

Use if var == value: / else: to include or exclude sections based on context. 

# access.mg
---
parameter: level (string): Access level of the user, e.g. "admin" or "user".
---
<< Access level: ${level} >>
if level == "admin":
  << You have full access. >>
else:
  << Read-only mode. >>

Loops

Loop over data just like in your programming languages.

# tasks.mg
---
description: A template that demonstrates looping over a list of tasks.
---
<< ## My Tasks >>
for task in tasks:
  << - ${task} >>

Includes

React like composability with includes.

# components/greeting.mg
---
description: A simple greeting component that can be included in other templates.
parameters: user (string): The name of the person to greet. (metadata is not required but good practice!)
---
<< Hello, ${user}! >>

# report.mg
---
description: A report template that includes a greeting component.
---
[[ components/greeting.mg user="Batman" ]]

Python API

Import Margarita directly into Python for agentic loop patterns — render templates, inject dynamic context, and pipe the output straight into your LLM calls.

from margarita.parser import Parser
from margarita.renderer import Renderer

metadata, nodes = Parser().parse(template)
renderer = Renderer(context={
    "task": "Summarize the key points",
    "context": "User is researching AI agents"
})

prompt = renderer.render(nodes)

Using Margarita with uv

Use uvx to run it as a one-off tool, or add it as a project dependency. Packages that ship .mg files can be used in includes.

# run margarita directly without installing
uvx margarita render greet.mg

# add margarita to your project
uv add margarita

# run via uv in a project that has margarita as a dependency
uv run margarita render greet.mg

# install globally as a uv tool
uv tool install margarita

# include a template shipped inside a uv-managed package
# report.mg
[[ my-prompt-package/summarize.mg topic="AI agents" ]]

Meet Lime

Lime is an extension that wraps your existing AI tools like Github Copilot / More soon. Create .mgx files and get surgical control over the context, tools, and state sent to them.

Guide: Build a Custom Ralph Wiggum Loop → Guide: Auto-Triage GitHub Issues → Guide: Spend less $ on Tokens →
Explore Lime Docs

Hello World

Your first .mgx file. Log a message, declare a state variable, write a prompt block, and run the agent — that's all it takes to get started with Lime.

---
description: Hello World!
---

// Log a message before the agent starts
@effect log "Welcome to Lime, let's get started!"

// Declare a state variable
@state variableA = "Hello World"

// Build the prompt
<< Why do we create ${variableA} programs? >>

// Run the agent
@effect run

Running the Agent

Use @effect run to execute the agent at any point in your .mgx file. Run it multiple times. Margarita renders all accumulated << >> blocks into the prompt before the agent is called.

# assistant.mgx
---
description: A simple Lime agent that answers a question.
---
// All statements are sequential

// 1. Add this to the context
<< You are a helpful assistant. What is the answer to life the universe and everything? >>

// 2. Run the agent
@effect run

// 3. Add more to the context
<< You need to think harder this time? >>

// 4. Run the agent again!
@effect run

Local Functions

Use @effect func to run a local Python function and store the return value in state — no LLM round-trip, no tokens consumed.

# processor.mgx
---
description: Call a Python function and store the result into state vars.
---
import summarize from module

// Run the summarize function and store into summary var
@effect func summarize("A really long text input") => summary

// Summary was stored into state vars so we can use it here!
<< Write a poem about the summary: ${summary} >>

@effect run

Dynamic Tools

Use @effect tools to add a tool function to your agent. Register only the tools you need. Use @effect tools clear to drop all tools.

# analyst.mgx
---
description: Fetch data with a tool, then pass it to the agent.
---
from modules import fetch, MyPydanticParams

// This registers your fetch Python function with the agent's tools.
// We specify the params with a pydantic model.
@effect tools fetch(params: MyPydanticParams) => data

<< You are a helpful assistant. You can fetch data from urls if you need... >>

@effect run
@effect tools clear

// Tools are cleared so no context is wasted on that tool.
<< Write a summary... >>
@effect run

Context Control

Use @effect context clear to reset the accumulated context between agent calls. Keeps each step focused and prevents context from ballooning across a multi-step workflow.

# multi-step.mgx
---
description: Two focused agent calls with a clean context boundary.
---
<< You are a code reviewer. >>
<< Review: ${code} >>

@effect run
@effect context clear

<< You are a documentation writer. >>
<< Write docs for: ${code} >>

@effect run

State

Values stored by @effect func and @effect tools persist for the lifetime of the run — templates and the agent can read from state at any point.

# stateful.mgx
---
description: Show how the agent state works.
---
@state count = 5
@state user = "Batman"

<<
The agent can get/set state variables using markdown.

Update the count variable to 10 and print out the user variable.
>>

@effect run

User Input

Use @effect input to pause execution and prompt the user for a value. The response is stored in state and available to the rest of the template.

# build-assistant.mgx
---
description: Prompt the user, then run the agent with their input.
---
// Pause and ask the user a question
@effect input "What do you want to build?:" => userInput

// Use the captured input in context
<< You are a helpful build assistant. >>
<< Help the user build: ${userInput} >>

@effect run

Memory

Use @memory var to declare a persistent variable that survives across runs. Values are read on startup and written back at the end — letting you build workflows that remember what happened last time.

# memory-example.mgx
---
description: Test Memory
---
// Declare a memory variable — loaded from persistent storage
@memory var valueA

<<
If valueA is not set, Set `valueA` to "This is a test of memory."
Otherwise write out it's set!
>>

@effect run
